# Signal Intelligence — Core Skills

This tool is designed to help users understand how the world works through events.
These skills are derived from how journalists construct explanations, but are reframed as cognitive operations rather than writing techniques.

## 1. Tension Detection

Where to look: understanding starts from contradictions, not information.

Definition:

- identify mismatches between price vs expectation
- identify mismatches between event vs narrative
- identify mismatches between data vs interpretation

In this case:

- highest YTD return vs lowest expected return

Tool implication:

- prioritize inputs that contain conflict signals
- suppress normal or already aligned information

## 2. Chronology Discipline

What happened, in what order: misordered facts lead to false explanations.

Definition:

- reconstruct what happened first
- reconstruct what changed
- determine whether events are causal, coincidental, or reinforcing

In this case:

- stock high and partnership announced on the same day
- causality is not yet established

Tool implication:

- explicitly model event timelines
- avoid causal claims without temporal support

## 3. Event Anchoring

Start from reality, not abstraction: understanding is grounded in specific events.

Definition:

- use concrete announcements
- use observable changes
- avoid abstract pattern-first reasoning

Tool implication:

- require a real-world anchor, either an event or a question, before explanation
- do not generate explanations from data alone

## 4. Mechanism Selection

What actually drives the outcome: understanding is choosing the right cause, not listing all causes.

Definition:

- from many possible explanations, select one to two dominant mechanisms

Failure mode:

- multiple-factors explanations

Tool implication:

- enforce mechanism sparsity
- reject explanations with too many drivers

## 5. Causal Chain Construction

How things connect: mechanisms must be expressed as processes, not labels.

Definition:

- build directional logic: cause -> intermediate -> outcome

In this case:

- RF complexity up -> testing difficulty up -> demand for integrated systems up

Tool implication:

- output must include step-by-step causal chains
- avoid static descriptions

## 6. Multi-Level Mapping

Where the explanation sits in the system: events operate across interacting layers.

Levels:

- company: what changed internally
- industry: structural drivers
- market: how expectations are priced

Failure mode:

- mixing unrelated layers
- staying at only one level

Tool implication:

- map explanation across levels only when relevant
- avoid redundancy

## 7. Concrete Representation

Can a human picture it: if you cannot visualize it, you do not understand it.

Definition:

- translate abstract terms into physical or operational processes

In this case:

- not RF testing solution
- but machines that test chips on the wafer before cutting

Tool implication:

- explanations must include concrete operational descriptions
- penalize vague language

## 8. Evidence Grounding

Why this explanation is valid: explanations must be tied to observable signals.

Definition:

- map each claim to data
- map each claim to documents
- map each claim to events

Failure mode:

- post-hoc narrative without support

Tool implication:

- require evidence links for each mechanism
- include counterevidence when present

## 9. Expectation vs Reality Updating

What actually changed: the world is understood through changes in expectations.

Definition:

- ask what was expected before
- ask what is implied now

In this case:

- analysts: low expected return
- market: strong performance

Tool implication:

- track prior belief vs updated belief
- highlight expectation gaps

## 10. Pattern vs One-Off Distinction

Is this structural or incidental: not all events are meaningful.

Definition:

- determine whether this is part of a broader trend
- determine whether this is an isolated event

In this case:

- partnership must be evaluated against RF demand growth
- partnership must be evaluated against FormFactor strategy

Tool implication:

- connect events to long-term patterns
- avoid over-weighting single data points

## 11. Information Sufficiency Awareness

When not to conclude: knowing when you do not know is part of understanding.

Definition:

- recognize missing analyst commentary
- recognize missing financial impact
- recognize missing adoption evidence

Tool implication:

- allow outputs like insufficient evidence
- allow outputs like needs more data
- prefer no explanation over a weak explanation

## System-Level Summary

These skills together define the tool’s purpose:

Transform events into structured understanding by identifying tensions, selecting minimal mechanisms, grounding them in evidence, and mapping them across interacting systems under uncertainty.
